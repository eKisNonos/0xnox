Quick Start Guide

Build and publish your first NONOS capsule in 6 steps.

Prerequisites

  • Ethereum wallet with ETH for gas fees
  • NOX tokens for marketplace pricing
  • Compiled ELF binary (x86_64 or ARM64)
  • Ed25519 keypair for cryptographic signing

Step 1: Generate Ed25519 Keys

Go to Key Management tab and click "Generate New Keypair". Your keys are stored locally in browser storage.

// Ed25519 Key Structure
Private Key (Seed): 32 bytes - KEEP SECRET
Public Key: 32 bytes - Embedded in manifest as dev_pubkey

Step 2: Configure Manifest

Fill out the manifest in Capsule Builder. The manifest defines your capsule's identity and permissions.

Step 3: Upload ELF Binary

Drag and drop your compiled binary. Supported: x86_64-elf, aarch64-elf. Max size: 50MB.

Step 4: Select Capabilities

Choose only the permissions your app needs. Users see these before unlocking.

Step 5: Build & Sign

Click "Build & Sign .noxc" to create the capsule binary with Ed25519 signature.

Step 6: Publish

Upload to IPFS, then register on CapsuleRegistry contract. Your capsule is now live!

Manifest Schema

The manifest is a JSON object embedded in every .noxc capsule. It defines metadata, permissions, and pricing.

Required Fields

{
  "id": "0x...",           // keccak256(name:version)
  "name": "com.dev.myapp", // Unique identifier
  "version": "1.0.0",      // Semantic version
  "dev_addr": "0x...",     // Developer ETH address
  "dev_pubkey": "base64",  // Ed25519 public key (32 bytes)
  "caps": "0x0000000000000001", // Capabilities bitmap (u64)
  "mem_min": 16,           // Minimum memory in MB
  "mem_max": 256,          // Maximum memory in MB
  "price": "1000000000000000000" // Price in NOX wei
}

Optional Fields

{
  "description": "App description",
  "category": "utility|game|defi|social|media|dev",
  "binary_sha256": "0x...",  // SHA256 of ELF binary
  "assets_sha256": "0x...",  // SHA256 of assets archive
  "homepage": "https://...", // Developer website
  "repository": "https://github.com/...",
  "created_at": 1704067200000 // Unix timestamp ms
}

Binary Format (.noxc)

NOXC (NOX Capsule) is the binary container format for NONOS applications. It combines manifest, executable, assets, and cryptographic signature.

File Structure

+------------------+
| Header (64 bytes)|
+------------------+
| Manifest (JSON)  |
+------------------+
| ELF Binary       |
+------------------+
| Assets (optional)|
+------------------+
| Signature (64 b) |
+------------------+

Header Format (64 bytes)

OffsetSizeFieldDescription
0x004magic0x43584F4E ("NOXC" LE)
0x042versionFormat version (0x0001)
0x062flagsReserved flags
0x084manifest_offsetOffset to manifest JSON
0x0C4manifest_sizeManifest size in bytes
0x104binary_offsetOffset to ELF binary
0x144binary_sizeBinary size in bytes
0x184assets_offsetOffset to assets (0 if none)
0x1C4assets_sizeAssets size (0 if none)
0x204sig_offsetOffset to Ed25519 signature
0x2428reservedReserved for future use

Magic Number

// Little-endian representation
NOXC_MAGIC = 0x43584F4E
           = 'N' | ('O' << 8) | ('X' << 16) | ('C' << 24)
           = [0x4E, 0x4F, 0x58, 0x43] in memory

ELF Binary Requirements

  • Architecture: x86_64 or aarch64
  • Format: Static ELF executable
  • Entry point: Standard _start or custom nonos_main
  • Max size: 50 MB
  • Linking: Static only, no dynamic libraries

Assets Archive

Optional tar.gz archive appended after the ELF binary. Contains static files (images, fonts, data).

// Assets structure
assets/
├── images/
│   └── logo.png
├── fonts/
│   └── main.ttf
└── data/
    └── config.json

// Max total size: 100 MB

Signing & Verification

All NONOS capsules must be cryptographically signed using Ed25519. The signature ensures integrity and authenticity.

Ed25519 Key Generation

// Generate 32-byte random seed
seed = crypto.getRandomValues(new Uint8Array(32))

// Derive keypair using TweetNaCl
keyPair = nacl.sign.keyPair.fromSeed(seed)

// Result:
// - privateKey: 64 bytes (seed + public key)
// - publicKey: 32 bytes (embedded in manifest as dev_pubkey)

Signing Process

// 1. Build capsule content (everything except signature)
content = header + manifest + binary + assets

// 2. Sign with Ed25519
signature = nacl.sign.detached(content, privateKey)
// Result: 64-byte signature

// 3. Append signature to capsule
capsule = content + signature

Verification Process

// Kernel verification steps:
1. Read header, validate magic (0x43584F4E)
2. Extract manifest, parse dev_pubkey
3. Read content (header + manifest + binary + assets)
4. Read 64-byte signature from end
5. Verify: nacl.sign.detached.verify(content, signature, publicKey)
6. If invalid, reject capsule with INVALID_SIGNATURE error

Key Storage Best Practices

  • Never share private keys - They sign on your behalf
  • Backup keys securely - Lost keys cannot be recovered
  • Use unique keys per project - Limits compromise impact
  • Export and store offline - Browser storage can be cleared

Key Rotation

To rotate keys, publish a new capsule version with the new public key. Users must re-unlock to use the new version.

Sandbox Security

NONOS executes capsules in a strict sandbox environment with multiple isolation layers.

Isolation Architecture

+------------------------+
|     User Space         |
|  +------------------+  |
|  | Capsule Process  |  |
|  | (sandboxed)      |  |
|  +------------------+  |
|          |             |
|  +------------------+  |
|  | Capability Guard |  |
|  +------------------+  |
|          |             |
+------------------------+
|   NONOS Kernel         |
|   - Syscall filtering  |
|   - Memory isolation   |
|   - Network policies   |
+------------------------+
|   Hardware / Host OS   |
+------------------------+

Security Boundaries

LayerProtectionEnforced By
ProcessMemory isolation, no shared stateNONOS Kernel
FilesystemVirtual FS, app-only accessVFS Layer

IPCExplicit capability requiredIPC Manager

Memory Limits

Each capsule runs within strict memory constraints defined in manifest:

mem_min: Guaranteed minimum allocation
mem_max: Hard limit, OOM kill if exceeded

// Kernel enforces:
- Stack size: 8 MB max
- Heap: Grows up to mem_max
- No memory mapping outside sandbox

Syscall Filtering

Only whitelisted syscalls are allowed. Attempting blocked syscalls results in immediate termination.

Allowed syscalls:
- read, write, close, fstat
- mmap, mprotect, munmap, brk
- nanosleep, clock_gettime
- exit, exit_group

Blocked (examples):
- fork, clone, execve
- ptrace, process_vm_*
- mount, umount
- reboot, init_module

Network Isolation

  • No network access without explicit capability
  • Outbound connections require CAP_NET_TCP_OUT or CAP_NET_UDP
  • Listening requires CAP_NET_TCP_LISTEN
  • DNS requires CAP_NET_DNS
  • All traffic logged for audit

Crash Recovery

If a capsule crashes or violates policy:

  1. Process terminated immediately
  2. Crash report generated (if opted-in)
  3. Resources cleaned up
  4. User notified with error code

Unlock Token System

The unlock system uses on-chain records to grant time-limited access to capsules.

Unlock Flow

User                    Contract                  NONOS Kernel
  |                         |                          |
  |-- unlockCapsule(id) --> |                          |
  |                         |-- transfer NOX --------> |
  |                         |-- split revenue -------> |
  |                         |-- create UnlockRecord -> |
  |<-- emit CapsuleUnlocked-|                          |
  |                         |                          |
  |------------------------ getUnlockRecord(id, user) |
  |                         |<-------------------------|
  |                         |---- return record ------>|
  |                         |                          |-- verify
  |                         |                          |-- download from IPFS
  |                         |                          |-- execute in sandbox

UnlockRecord Structure

struct UnlockRecord {
    bytes32 token;         // Unique unlock token
    uint64 timestamp;      // Block timestamp of unlock
    uint64 expiresAt;      // timestamp + 86400 (24 hours)
    bytes32 manifestHash;  // Hash of manifest at unlock time
    bytes32 approvedCaps;  // Capabilities granted
}

// Storage mapping
mapping(uint256 => mapping(address => UnlockRecord)) unlocks;
// unlocks[capsuleId][userAddress] = UnlockRecord

Token Generation

// On-chain token generation (Solidity)
token = keccak256(abi.encodePacked(
    capsuleId,
    msg.sender,
    blockhash(block.number - 1),
    block.timestamp,
    nonce++
));

// Token is unique per unlock, cannot be predicted

Expiration & Renewal

  • Duration: 24 hours from unlock
  • Renewal: Call unlockCapsule again (full price)
  • Grace period: None - expires exactly at expiresAt
  • Refunds: Not supported (blockchain is immutable)

Kernel Verification

// NONOS kernel verification steps:
1. Query: getUnlockRecord(capsuleId, wallet)
2. Check: record.token != bytes32(0)           // Has unlocked
3. Check: block.timestamp < record.expiresAt   // Not expired
4. Check: hash(manifest) == record.manifestHash // Same version
5. Grant: capabilities = record.approvedCaps
6. Execute: run capsule in sandbox with granted caps

Bonding Curve Mechanics

Every token launched on 0xNOX uses an exponential bonding curve that determines price based on supply:

Price = BasePrice * e^(k * Supply)
BasePriceInitial token price (0.0000001 ETH)
kCurve steepness constant
SupplyCurrent circulating supply

Key Properties

  • Instant Liquidity: Buy and sell anytime against the curve
  • Price Discovery: Fair, transparent pricing mechanism
  • No Rug Pulls: Liquidity locked in contract until graduation
  • Predictable: Price changes are deterministic

Creating a Token

Requirements

  • Connected wallet with ETH for gas
  • 10,000 NOX creation fee (free with ZeroState Pass)
  • Token name, symbol, and optional metadata

Token Creation Process

  1. Fill in token details - Name, symbol, description
  2. Upload branding - Logo (256x256) and banner (1200x400)
  3. Add social links - Twitter, Discord, Telegram, website
  4. Submit transaction - Approve NOX (if needed) and create
  5. Token is live - Instantly tradeable on bonding curve
💡 The creation fee is split: 50% is burned forever, 50% goes to NOX stakers. This creates deflationary pressure while rewarding the community.

Trading on Bonding Curves

Every token has a bonding curve that provides instant liquidity:

Buying Tokens

  1. Select a token from the live list
  2. Enter ETH amount to spend
  3. Review estimated tokens received
  4. Click "Buy" and confirm transaction

Selling Tokens

  1. Go to the token page
  2. Enter token amount to sell
  3. Review ETH you'll receive
  4. Click "Sell" and confirm transaction
⚠️ Slippage can occur on large trades. The bonding curve price changes with each trade. Always check estimated amounts before confirming.

Wallet Setup

Option 1: Browser Wallet (Recommended)

Use MetaMask, Rabby, or any Web3 wallet:

  1. Install MetaMask extension
  2. Create or import a wallet
  3. Click "Connect" on 0xNOX
  4. Approve the connection

Option 2: WalletConnect

Connect mobile wallets via QR code:

  1. Click "Connect" then "WalletConnect"
  2. Scan QR code with mobile wallet
  3. Approve connection in your app

Option 3: Generate Wallet

Create a wallet instantly, no extensions needed:

  1. Click "Connect" then "Generate Wallet"
  2. Save your recovery phrase securely
  3. Click "Use This Wallet"
  4. Fund with ETH via deposit address

Bridging NOX

Transfer NOX between Ethereum and Cellframe networks:

Bridge to Cellframe

  1. Go to Bridge page
  2. Enter NOX amount (min 100)
  3. Enter or generate Cellframe address
  4. Click "Bridge to Cellframe"
  5. Approve NOX spend (first time only)
  6. Confirm bridge transaction
  7. Wait for 3 validator confirmations

Cellframe Wallet

Generate a Cellframe wallet directly from the bridge:

  • Click "Generate" next to address field
  • Save your private key immediately
  • Address auto-fills for bridging
  • Use address to receive CF-20 NOX

Frequently Asked Questions

Is this on mainnet?

Yes, 0xNOX is deployed on Ethereum mainnet. All transactions use real ETH and real tokens.

What happens at graduation?

At $69,420 market cap, the token automatically graduates. ETH and tokens are sent to Uniswap, LP tokens are burned, and the token becomes freely tradeable on DEXes.

Can creators rug pull?

No. All liquidity is locked in the bonding curve contract until graduation. There are no admin keys or special permissions that allow draining funds.

How do I get NOX tokens?

NOX is available on Uniswap. You can also earn NOX through staking rewards or by creating popular tokens on the platform.

What is ZeroState Pass?

ZeroState Pass is an NFT that grants free token creation on 0xNOX, plus boosted staking rewards and exclusive features.
